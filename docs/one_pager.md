# One Pager
## LinkMap

**한 줄 소개**  
인스타그램, 유튜브, 블로그에서 발견한 장소 링크를 붙여 넣으면 장소 정보를 반자동으로 정리하고, **저장 이유**와 **출처**를 함께 기록해 **지도에서 다시 탐색**할 수 있게 해주는 일상형 장소 저장 서비스

---

## 1. 개요
사람들은 카페, 식당, 전시, 여행지 등 가보고 싶은 장소를 인스타그램, 유튜브, 블로그 같은 다양한 플랫폼에서 발견한다. 하지만 대부분은 각 플랫폼 내부의 저장 기능에 의존하거나, 필요할 때 지도 앱에 직접 다시 입력하는 방식으로 관리한다. 이 과정에서 정보는 여러 플랫폼에 흩어지고, 시간이 지나면 **왜 저장했는지조차 기억하기 어려워진다.**

**LinkMap**은 이러한 불편을 해결하기 위해, **링크 붙여 넣기만으로 장소 정보를 반자동 저장**하고, **저장 이유·출처·카테고리·방문 상태**를 함께 기록하여 **지도 기반으로 다시 탐색**할 수 있게 하는 서비스를 목표로 한다.

---

## 2. 문제 정의
사용자들은 다양한 플랫폼에서 장소 정보를 저장하지만, 실제로 다시 활용하는 과정에서는 다음과 같은 불편을 겪는다.

- **정보 분산**: 인스타그램, 유튜브, 블로그, 지도 앱 등에 저장 정보가 흩어져 있어 한 번에 관리하기 어렵다.
- **맥락 손실**: 시간이 지나면 **왜 저장했는지**, **어디서 보고 저장했는지**, **가보려는 곳인지 / 이미 가본 곳인지** 기억하기 어렵다.
- **지도 기반 탐색 부재**: 사용자가 실제로 원하는 것은 목록이 아니라 **지도에서 위치 중심으로 저장 장소를 다시 보는 경험**이지만, 기존 저장 방식은 이를 충분히 지원하지 못한다.
- **실행 연결 부족**: 저장은 많이 하지만, 다시 찾기 어렵고 실제 방문 계획으로도 잘 이어지지 않는다.

**결과적으로**, 사용자는 많은 장소를 저장해두고도 다시 찾기 어렵고, 저장 데이터가 일상 속 활용으로 이어지지 못한다.

---

## 3. 문제의 근본 원인
이 문제는 단순히 저장 기능이 부족해서가 아니라, 기존 저장 구조 자체가 **콘텐츠 소비 중심**이기 때문에 발생한다.

- **플랫폼 저장 기능의 한계**: 인스타그램과 유튜브의 저장 기능은 게시물/영상을 다시 보기 위한 목적에 가깝고, 장소 중심 정리에는 적합하지 않다.
- **지도 앱의 한계**: 지도 앱은 위치 저장에는 강하지만, **저장 이유 / 출처 링크 / 개인 메모** 같은 맥락 정보를 함께 관리하기 어렵다.
- **통합 관리 도구 부재**: 여러 플랫폼에서 발견한 장소 정보를 **지도 기반으로 통합해 관리하는 개인화 도구**가 부족하다.

즉, 현재의 방식은 **“발견 순간의 저장”** 은 편하지만, **“나중에 다시 꺼내 쓰는 경험”** 까지는 충분히 고려하지 못하고 있다.

---

## 4. 해결책
### 서비스 관점
사용자가 인스타그램, 유튜브, 블로그 링크를 붙여 넣으면 장소 정보를 반자동으로 정리하고, **저장 이유·출처·카테고리·방문 상태**를 함께 저장해 **지도에서 다시 탐색**할 수 있도록 한다.

### 기술 관점
이를 위해 백엔드에서는 다음 흐름을 구현한다.

- **플랫폼별 URL 판별**
- **링크 메타데이터 추출**
- **장소 후보 검색 및 매칭**
- **사용자 확인 후 저장**
- **지도 범위 기반 조회 및 카테고리 필터링**

### 핵심 사용자 경험
- 링크 붙여 넣기만으로 장소 등록 시작
- 장소 후보 확인 후 반자동 저장
- 지도 위에서 저장 장소 핀 확인
- 카테고리별 필터링
- 상세 화면에서 저장 이유 / 출처 / 원문 바로가기 확인
- 지도 앱으로 길찾기 연결

---

## 5. 가설
이 프로젝트는 다음과 같은 가설을 바탕으로 한다.

- **반자동 링크 등록**을 제공하면, 사용자는 지도 앱에 직접 장소를 다시 검색·입력하는 부담 없이 더 쉽게 저장할 것이다.
- **저장 이유와 출처**를 함께 남기면, 사용자는 시간이 지나도 저장 맥락을 더 잘 회상할 것이다.
- **지도 기반 조회**를 제공하면, 사용자는 특정 지역의 저장 장소를 목록보다 더 빠르게 탐색할 것이다.
- **방문 상태 관리**를 제공하면, “가보고 싶은 곳”과 “이미 가본 곳”을 구분할 수 있어 저장 데이터의 활용도가 높아질 것이다.

---

## 6. 기회비용 / 트레이드오프
이 프로젝트는 사용 편의성과 자동화를 높이는 대신, 다음과 같은 비용과 제약을 가진다.

- **구현 복잡도 증가**: 플랫폼마다 링크 구조와 메타데이터 형식이 달라 일관된 처리 로직 설계가 필요하다.
- **외부 API 의존성**: 지도/장소 API에 의존하므로 호출 제한, 비용, 응답 품질을 고려해야 한다.
- **완전 자동화의 한계**: 자동 추출 정확도가 낮을 수 있어, 초기에는 **사용자 확인 기반 반자동 저장**이 더 현실적이다.
- **MVP 범위 제한 필요**: 초기에는 지원 플랫폼과 자동화 범위를 제한해야 완성도 있는 결과를 만들 수 있다.

---

## 7. 기대 결과
### 사용자 관점
- 흩어진 플랫폼 속 장소 정보를 **하나의 지도 위에서 통합 관리**할 수 있다.
- 저장 이유와 출처를 함께 남겨 **왜 저장했는지 쉽게 회상**할 수 있다.
- 저장 데이터가 단순 목록이 아니라, **실제 방문과 이동으로 이어지는 도구**가 된다.

### 개발자 관점
이 프로젝트를 통해 다음과 같은 **자바 백엔드 역량**을 보여줄 수 있다.

- **도메인 설계**
- **URL 처리 및 플랫폼별 분기 로직 설계**
- **외부 지도 API 연동**
- **지도 조회 API 설계**
- **예외 처리 및 테스트 코드 작성**
- **실사용 문제를 해결하는 서비스형 포트폴리오 구현**

---

## 8. MVP 범위
초기 MVP는 아래 기능에 집중한다.

- 링크 붙여 넣기
- 링크 메타데이터 분석 및 장소 후보 추출
- 사용자 확인 후 장소 저장
- 카테고리 / 저장 이유 / 출처 링크 / 방문 상태 저장
- 지도 기반 저장 장소 조회
- 카테고리별 필터링
- 상세 정보 조회
- 원문 바로가기 및 길찾기 연결

---

## 9. 핵심 가치
- **통합**: 여러 플랫폼에 흩어진 장소 정보를 한곳에서 관리
- **맥락 보존**: 왜 저장했는지, 어디서 본 건지 함께 기록
- **지도 기반 재탐색**: 목록이 아닌 위치 중심으로 다시 확인
- **실행 연결**: 원문 이동과 길찾기를 통해 실제 방문으로 연결
