# ADR-007: 장소 추출 전략 — LLM 단독 방식 채택

- 날짜: 2026-03-18
- 상태: 결정됨 (구현 예정)

---

## 배경

poc 프로젝트에서 규칙 기반 추출기(RuleBasedPlaceCandidateExtractor)로
YouTube 리스트형 콘텐츠 Top1 정확도 86.67%를 달성했다.

그러나 규칙 기반은 다음 한계를 가진다:
- 새로운 구조화 패턴이 생길 때마다 수동 규칙 추가 필요
- 문맥 이해 없음 ("성수 카페 3곳 추천" vs "성수 카페 방문 후기" 구분 어려움)
- 노이즈 단어 목록 수동 관리

세 가지 선택지를 검토했다:
1. GLiNER-ko 단독 (NER 모델)
2. LLM 단독
3. GLiNER-ko + LLM 하이브리드

---

## 결정사항

**LLM 단독 방식**을 채택한다.

```
본문(제목 + 설명 + 자막/본문)
  → LLM (후보 추출 + 콘텐츠 분류 + 검색 쿼리 생성)
  → Kakao 장소 매칭
  → topCandidates[] 반환
```

LLM이 담당하는 역할:
- 장소 후보명 추출 (rawName, normalizedName)
- 지역/카테고리 힌트 부착
- `SINGLE_PLACE` / `MULTI_PLACE` / `REVIEW_REQUIRED` 분류
- Kakao 검색용 query expansion 생성
- 응답은 구조화된 JSON으로 강제 (Structured Output)

---

## 대안

### A. GLiNER-ko 단독
- 장점: 빠름, 로컬, API 비용 없음
- 단점: 문맥 이해 없음, SINGLE/MULTI 분류 불가
- 탈락 이유: 분류 정확도 낮을 것으로 예상

### B. GLiNER-ko + LLM 하이브리드
- 장점: 토큰 절약, 각자 역할 분담
- 단점: Python 서비스 분리 필요, 아키텍처 복잡도 증가
- 탈락 이유: 구현 복잡도 대비 현시점에서 불필요한 최적화

### C. 규칙 기반 유지 (poc 이식)
- poc 코드를 그대로 이식하는 방법
- 탈락 이유: 규칙 관리 부담, 확장성 한계

---

## 결과

### 긍정적
- 구현 단순 — 프롬프트 수정으로 동작 개선 가능
- 새로운 플랫폼/콘텐츠 형태 자동 대응
- SINGLE/MULTI 분류 정확도 기대치 높음
- poc 대비 규칙 유지보수 부담 없음

### 부정적
- LLM API 호출 비용 발생 (요청당 과금)
- 응답 지연 증가 (평균 1-3초 추가 예상)
- 긴 유튜브 자막은 토큰 비용 높음 → 요약/청크 전략 필요
- Structured Output 파싱 실패 시 fallback 처리 필요

### 중립적
- 트래픽이 늘면 GLiNER-ko 하이브리드로 전환 고려 (ADR 재검토)
- 프롬프트는 버전 관리 필요 (코드와 함께 커밋)
